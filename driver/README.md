# Network Deck Controller — driver

Kernel driver presenting a virtual Steam Deck controller to Windows via the
USB Device Emulation (UDE) class extension. User-mode (`client-win`) feeds
HID reports in over an IOCTL; the driver hands them to the virtual USB
interrupt-IN endpoint; Steam Input sees a Deck.

**This is a scaffold.** It will not build out of the box — there's no
Visual Studio project file (`.vcxproj`) on purpose. You set that up once
locally against your installed WDK, then drop these files into it.

## Layout

```
driver/
  inc/
    public.h                    Driver/user-mode contract (IOCTLs, GUID).
                                Mirrored by the Rust client.
  src/
    common.h                    Kernel-side shared decls + per-device context.
    driver.cpp                  DriverEntry + EvtDriverDeviceAdd.
    usbdevice.cpp               Virtual USB device + descriptors + plug-in.
    queue.cpp                   IOCTL queue (push input, pend output).
  inf/
    NetworkDeckController.inf   Installation INF (root-enumerated).
```

## Prerequisites

- **Visual Studio 2022** (Community is fine), with the "Desktop development
  with C++" workload.
- **Windows SDK** matching your target Windows build.
- **Windows Driver Kit (WDK)** — must match your SDK version exactly. Get
  it from <https://learn.microsoft.com/en-us/windows-hardware/drivers/download-the-wdk>.
- **Spectre-mitigated MSVC libraries** (the WDK installer will offer to
  install these).

## Setting up the project (one-time, manual)

The straightforward path:

1. Clone Microsoft's reference samples:
   ```
   git clone https://github.com/microsoft/Windows-driver-samples.git
   ```
2. Open one of the UDE samples (`usb/UDE_*`) in VS to confirm your WDK
   install can build a UDE driver. **Verify the sample builds before
   touching ours** — saves hours.
3. Create a new project in VS: *File → New → Project →* "Kernel Mode
   Driver, Empty (KMDF)". Place it at `driver/NetworkDeckController/`.
4. **Add UDE support to the project:**
   - Project Properties → Linker → Input → Additional Dependencies → add
     `UdecxKmdf.lib`.
   - Project Properties → C/C++ → General → Additional Include Directories
     → add `$(WDKContentRoot)Include\$(WindowsTargetPlatformVersion)\km`
     and the `Udecx.h` location (usually already covered).
5. Add our files to the project:
   - `inc/public.h`
   - `src/common.h`
   - `src/driver.cpp`
   - `src/usbdevice.cpp`
   - `src/queue.cpp`
   - `inf/NetworkDeckController.inf` (right-click project → Add → Existing
     Item — VS will treat it as the driver INF).
6. Project Properties → Driver Signing → Sign Mode → "Test Sign", and
   either "Sign as company" or attach a self-signed test cert.
7. Build for x64 Release (or Debug for kernel debugging).

Output: `NetworkDeckController.sys`, `.inf`, `.cat` in the build directory.

## Test signing on the dev machine

Until you have an EV cert + Microsoft Partner Center attestation signing,
your driver isn't trusted by Windows. One-time setup on the **install
target** (your dev machine):

```powershell
# Run as administrator, then reboot.
bcdedit /set testsigning on
```

You'll see a "Test Mode" watermark on the desktop. Reverse with
`bcdedit /set testsigning off`.

For the **build machine** (also your dev machine usually), import the test
cert generated by VS into `Trusted Root Certification Authorities` and
`Trusted Publishers`. The WDK docs walk through this; once done you won't
need to import again per build.

## Install / uninstall

```powershell
# Install (from the directory containing the .sys, .inf, .cat)
pnputil /add-driver NetworkDeckController.inf /install

# After install, find the device-instance ID
pnputil /enum-devices /class System | Select-String "NetworkDeck"

# Uninstall by oem*.inf id (find it under pnputil /enum-drivers)
pnputil /delete-driver oem15.inf /uninstall
```

## What the scaffold does and does not do

**Does:**
- Establish driver/user-mode IOCTL contract (`inc/public.h`).
- Establish the WDF + UDE skeleton (DriverEntry, EvtDriverDeviceAdd,
  device interface registration, IOCTL queue, manual queue for pended
  output reports).
- Provide a working INF.

**Does not (TODO):**
- USB descriptors are placeholder bytes. Capture from a real Deck on
  Windows (USBView, UVCView in WDK, or `Get-PnpDeviceProperty`) and
  replace the byte arrays in `usbdevice.cpp`.
- HID report descriptor — same.
- `UDECX_WDF_DEVICE_CONFIG_INIT` callback set is stubbed; Microsoft's UDE
  sample shows the right initializer pattern with current WDK headers.
- `UsbDeviceCreate` body is `STATUS_NOT_IMPLEMENTED`. Real implementation
  goes through `UdecxUsbDeviceInitAllocate` →
  `UdecxUsbDeviceInitAddDescriptor` (×N) → `UdecxUsbDeviceCreate` →
  `UdecxUsbEndpointCreate` (×N) → `UdecxUsbDevicePlugIn`.
- Input report path: routing bytes from the IOCTL to a queued endpoint
  request is a TODO inside `EvtIoDeviceControl`.
- Output report path: the manual queue exists but nothing completes
  parked requests yet. The completion site is wherever
  `EvtUsbEndpointReady` (interrupt-OUT) or feature-set callback fires
  inside `usbdevice.cpp`.

The build sequence in `../ARCHITECTURE.md` calls these out in order:

> 3. Static UDE driver: hardcoded descriptors, canned report on a timer.
>    Goal: Steam shows "Steam Deck Controls."
> 4. Feature-report path: lizard-mode disable, haptics-config ack.
> 5. User-mode IPC + live HID frames over IOCTL.

Step 3 is where you'd first see the device appear in Windows. Don't try
to wire up user-mode IPC before the static device shows up — you'll be
debugging two layers at once.
